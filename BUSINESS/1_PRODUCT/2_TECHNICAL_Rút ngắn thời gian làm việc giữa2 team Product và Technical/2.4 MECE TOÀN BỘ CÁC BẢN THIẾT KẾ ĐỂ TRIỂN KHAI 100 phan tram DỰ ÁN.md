## PHẦN III: THIẾT KẾ KIẾN TRÚC PHẦN MỀM (Architecture Design)

### III.1. System Architecture Diagrams (3-4 sơ đồ)

- **Conceptual Architecture Diagram**:
  ```
  User Browser
       ↓
  finAI Chrome Extension
       ↓
  Python Backend (Flask/FastAPI)
       ↓
  GPT-4 API + Financial Doc Parser
       ↓
  EDGAR, Yahoo Finance, Company Websites, News APIs
       ↓
  PostgreSQL Database + Redis Cache
  ```

- **Component Architecture Diagram**:
  - Extension Component (content script, background service worker)
  - Backend Components (auth service, document parser, AI service, compliance service)
  - Database (users, documents, analyses, portfolios)
  - External integrations (GPT-4, broker APIs, data providers)

- **Deployment Architecture Diagram**:
  - Frontend: Chrome/Edge extensions (user device)
  - Backend: Python server on Vercel or AWS
  - Database: PostgreSQL on managed service (AWS RDS)
  - Cache: Redis (optional, for performance)
  - CDN: For static assets

- **Data Flow Diagram (DFD)**:
  - Level 0: User analyzes document → finAI processes → User gets insights
  - Level 1: 
    - User uploads document
    - Extension sends to backend
    - Backend extracts text, calls GPT-4
    - GPT-4 returns structured data
    - Backend validates, stores in DB
    - Returns to extension for display

### III.2. API Design (1 tài liệu)

- **REST API Specification**:
  - Endpoints (with HTTP methods, paths, description)
  - Request/response schemas (JSON examples)
  - Authentication (bearer token, JWT)
  - Rate limiting (requests/minute)
  - Error handling (error codes, messages)
  - Pagination (for lists)
  
  Examples:
  ```
  POST /api/v1/analyses
  Request: { documentText: "...", ticker: "AAPL" }

  Response: { ...full analysis object }
  
  POST /api/v1/portfolios
  Request: { positions: [...], watchlist: [...] }
  Response: { portfolioId: "456", summary: {...} }
  ```

### III.3. Database Design (2 tài liệu)

- **Entity-Relationship Diagram (ERD)**:
  - Users table (id, email, subscription_tier, created_at)
  - Documents table (id, user_id, file_name, content, uploaded_at)
  - Analyses table (id, user_id, document_id, metrics, risks, created_at)
  - Portfolios table (id, user_id, positions, watchlist)
  - Snapshots table (id, analysis_id, pdf_content, exported_at)
  - Audit_logs table (id, user_id, action, resource, timestamp)

- **Database Schema (SQL)**:
  ```sql
  CREATE TABLE users (
    id UUID PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255),
    subscription_tier ENUM('free', 'pro', 'enterprise'),
    created_at TIMESTAMP,
    updated_at TIMESTAMP
  );
  
  CREATE TABLE documents (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    file_name VARCHAR(255),
    content TEXT,
    document_type ENUM('10k', 'earnings', 'news', 'other'),
    uploaded_at TIMESTAMP
  );
  
  CREATE TABLE analyses (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    document_id UUID REFERENCES documents(id),
    metrics JSONB,  -- { revenue, ebitda, debt, margin, ... }
    risks JSONB,    -- [{ type, severity, description, source }, ...]
    confidence_score FLOAT,
    created_at TIMESTAMP
  );
  ```

### III.4. Architecture Patterns (1 tài liệu)

- **Design Patterns Used**:
  - MVC (Model-View-Controller): Separation of concerns
  - Factory Pattern: Document parser factory (create different parsers for different doc types)
  - Strategy Pattern: AI model selection (GPT-4 vs Claude vs fine-tuned model)
  - Observer Pattern: Real-time updates to portfolio monitoring
  - Singleton Pattern: Database connection pool, API client
  - Repository Pattern: Data access abstraction
  - Decorator Pattern: Add validation, logging, error handling to API responses

- **Architecture Style Rationale**:
  - Why browser extension? (distributed computing, works offline after first load)
  - Why serverless backend? (scalable, pay-per-use, fast deployment)
  - Why PostgreSQL? (ACID compliance, JSONB for flexible schema, mature)
  - Why GPT-4? (best-in-class extraction, fine-tunable, commercially available)

## PHẦN IV: THIẾT KẾ CHI TIẾT (Detailed Design)

### IV.1. Component Specifications (5-10 tài liệu)

- **Authentication Component**:
  - Sign up flow (email, password, email verification)
  - Login flow (email, password, or OAuth with Google/GitHub)
  - Token management (JWT, refresh tokens, expiration)
  - Password reset flow
  - Logout & session management

- **Document Parser Component**:
  - Text extraction from PDF/HTML/plain text
  - Chunking strategy (split large documents into sections)
  - Noise removal (headers, footers, page numbers)
  - Structure preservation (sections, tables)
  - Language support (English first, later expand)

- **AI Analysis Component**:
  - Prompt engineering for metrics extraction
  - Multi-step reasoning (understand context → extract metrics → flag risks)
  - Confidence scoring (based on source text proximity, keyword matching)
  - Hallucination detection (compare against multiple sources, flag conflicts)
  - Validation layer (compare extracted numbers against secondary sources)

- **Risk Flagging Component**:
  - Rule-based patterns (leverage > 3x, covenant breach keywords)
  - ML-based anomaly detection (unusual changes in metrics)
  - Severity scoring (critical, high, medium, low)
  - Risk taxonomy (leverage, covenant, guidance, concentration, etc.)

- **Snapshot Generator Component**:
  - Layout design (header, key metrics table, risks table, footer)
  - PDF generation (using Puppeteer or ReportLab)
  - Includes source attribution (page numbers, quote snippets)
  - Exportable formats (PDF, PNG, JSON)

- **Portfolio Monitoring Component** (v1.5):
  - Watchlist management (add/remove tickers)
  - News aggregation (real-time alerts)
  - Filing monitoring (when 10-K/8-K released)
  - Change detection (metrics updates, new risks)
  - Email digest generation

### IV.2. Algorithm Design (2-3 tài liệu)

- **Metric Extraction Algorithm**:
  ```
  1. Split document into sections (using headers, page breaks)
  2. For each target metric (revenue, ebitda, debt, etc.):
     a. Query GPT-4: "Find revenue in this section"
     b. Get response: { value, unit, confidence, source_snippet }
     c. Validate: Check if value is reasonable for company size
     d. Cross-check: Look for same metric in other sections
     e. Store: { metric_name, value, confidence, source_page }
  3. Return: Array of metrics with confidence scores
  ```

- **Risk Flagging Algorithm**:
  ```
  1. For each identified metric:
     a. Check against thresholds (leverage > 3x = red flag)
     b. Compare to historical averages (25% YoY decline = anomaly)
     c. Check against industry benchmarks
  2. For text analysis:
     a. Keyword matching: "covenant breach", "material weakness", "discontinued ops"
     b. Sentiment analysis: Negative tone in risk factors section
  3. For each risk found:
     a. Assign severity (critical, high, medium, low)
     b. Record source location (page, quote)
     c. Assign risk category (leverage, liquidity, operational, etc.)
  4. Return: Sorted risk list with confidence scores
  ```

- **Document Classification Algorithm**:
  - Identify document type: 10-K, 10-Q, earnings transcript, news article, broker report
  - Parse accordingly: Different parsers for different document structures
  - Classify company industry: Technology, healthcare, financial services, etc.
  - Extract metadata: Filing date, company name, ticker, fiscal period

### IV.3. Security & Compliance Design (2 tài liệu)

- **Security Architecture**:
  - Authentication: JWT tokens with 15min expiration + refresh tokens (7 days)
  - Authorization: Role-based access (free, pro, enterprise, admin)
  - Encryption: TLS/SSL for all traffic, AES-256 for data at rest
  - API security: Rate limiting (100 req/min per user), CORS restrictions
  - Secrets management: Environment variables, AWS Secrets Manager
  - Input validation: Sanitize all user inputs to prevent injection attacks
  - OWASP compliance: Follow OWASP Top 10

- **Compliance & Audit Design**:
  - Audit logging: Every API call logged with timestamp, user_id, action, resource, result
  - Source attribution: Every number includes page reference, confidence score, source snippet
  - Data residency: Store user data in single region (US, EU based on user choice)
  - Retention policy: Delete analyses after 90 days (configurable by user)
  - GDPR/CCPA compliance: Right to delete, data portability, consent tracking
  - Disclaimer framework: Every output includes: "This is research, not investment advice"
  - Legal positioning: Market as "research tool", NOT "investment advisor"

### IV.4. Performance & Scalability Design (1 tài liệu)

- **Performance Targets**:
  - Time to first byte: < 1 second
  - Analysis generation: < 60 seconds for most documents
  - Page load: < 2 seconds (sidebar extension)
  - API response time: < 500ms (p95)

- **Scaling Strategy**:
  - Database: Horizontal sharding by user_id (separate databases per user cohort)
  - Caching: Redis for frequently accessed data (user profiles, metrics cache)
  - API: Stateless backend, can scale to multiple instances
  - Rate limiting: Per-user quotas (free: 5/month, pro: unlimited)
  - CDN: Serve static assets (JS, CSS, fonts) from Cloudflare

- **Load Testing Plan**:
  - Target: 10,000 concurrent users
  - Simulate: 100 analyses/second
  - Success criteria: < 2 sec latency at p95, < 1% error rate

### VII.1. Viral & Growth Loops (1 tài liệu)

  - User generates snapshot → Includes "Powered by finAI" footer → Recipient clicks → New user signup
  - Metrics: Invite rate (% users who share), Acceptance rate (% recipients who signup), K-factor (viral coefficient)
  - Target: K-factor 0.5 (each user brings 0.5 new user)

- **Content Loop** (Secondary):
  - User makes snapshot public → Indexed by Google → SEO traffic → Signup
  - Metrics: Public snapshots created/month, Organic traffic sessions, CVR from organic

- **Referral Loop** (Tertiary):
  - User invites analyst friend → Friend signs up → Original user gets 1 month free Pro
  - Metrics: Referral rate, acceptance rate, LTV lift from referrals

### VII.2. Retention & Monetization (1 tài liệu)

- **Aha Moment** (Activation):
  - User completes 3 analyses in first week → Sets up portfolio monitoring
  - Metric: % users who reach aha moment by D7

- **Habit Loop**:
  - User researches 1 stock/week → Email digest on weekends → User opens email → Returns to finAI
  - Frequency: Email sent weekly, 7-day active user target

- **Monetization Triggers**:
  - Free tier: 5 analyses/month → Nudge at 4/5 complete
  - Pro upsell: Portfolio monitoring only in Pro → Unlock at signup
  - Enterprise: Admins can see 10+ users → Outreach from sales team

## TỔNG KẾT

**Tổng cộng: 60+ thiết kế/tài liệu**

Nếu 1 người + 1 AI (code-generation AI như Claude hoặc ChatGPT) có tất cả các thiết kế này, có thể:
1. **Hiểu rõ** vấn đề khách hàng, thị trường, chiến lược
2. **Thiết kế** sản phẩm chi tiết từ UI đến database
3. **Lập trình** backend, frontend, bao gồm tests
4. **Triển khai** hệ thống, monitoring, support
5. **Mở rộng** với growth loops, partnership, fundraising

**Thời gian ước tính**: 18-24 tháng từ khởi động đến Product-Market Fit (với 1 người full-time + 1 AI 24/7)

**Điểm cân bằng**: Nếu thiết kế chi tiết thì code dễ hơn; nếu không có thiết kế thì phải code rồi refactor nhiều lần.
